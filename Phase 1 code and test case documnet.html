2. Test Area 1 – SQL Injection Fix

File: UserManagementDaoImpl.java

What was changed:
The changeUserPassword() method previously built SQL queries by directly concatenating the username and password values into the query string. This allowed an attacker to inject malicious SQL code through the input fields.

Before (vulnerable):
"select password from users where username = '" + username + "'"

After (fixed):
session.createNativeQuery("SELECT password FROM users WHERE username = :username").setParameter("username", username)

How it works: The database driver treats the value of username as pure data, never as SQL syntax. Even if an attacker types ' OR '1'='1, the database will look for a user literally named ' OR '1'='1 and find none.

3. Test Area 2 – Password Logging Removed

File: CommonUtil.java – getPasswordBcrypt()

What was changed:
The method previously logged the raw plaintext password to the log file before hashing it.

Before (vulnerable):
logger.info("getPasswordBcrypt password :" + password);

After (fixed):
The line above was removed entirely.

How it works: Without the logger line, the password is never written to any log file. Even if an attacker gains read access to the log file (onboard.log), they will find no passwords there.

4. Test Area 3 – MD5 Method Removed

File: CommonUtil.java – getPasswordMD5()

What was changed:
The getPasswordMD5() method was removed entirely from CommonUtil.

Why MD5 is dangerous for passwords: MD5 hashes are extremely fast to compute — a modern GPU can check billions of MD5 hashes per second. This makes MD5-hashed passwords trivially crackable using brute force or rainbow tables. BCrypt is intentionally slow and includes a salt, making it the correct choice.

After (fixed): The method no longer exists. All password hashing uses BCryptPasswordEncoder.

5. Test Area 4 – Race Condition Fix (Moodle Controller)

File: MoodleController.java

What was changed:
Three instance-level variables were removed from the controller class:
private String absolutePath;
private HttpSession session;
private String role_id;

Why this is a problem: Spring creates only ONE instance of each controller and shares it across all HTTP requests. If 100 users are logged in at the same time, they all share the same controller object. If User A's request sets role_id = "ADMIN" and then User B's request immediately changes it to role_id = "VIEWER", User A's code may now execute with the wrong role — this is a race condition and a privilege escalation risk.

After (fixed): session and role_id are declared as local variables inside each method. Local variables exist only for the duration of that single request and are never shared.

6. Test Area 5 – Path Traversal Prevention

File: MoodleController.java – downloadFile() and downloadFiles()

What was changed:
Two layers of protection were added to the downloadFile() and downloadFiles() methods:

Layer 1 – Regex check:
private static final Pattern SAFE_FILENAME = Pattern.compile("[\\w\\-\\.]+");
Only letters, digits, underscores, hyphens, and dots are allowed. Any other character immediately returns HTTP 400.

Layer 2 – Canonical path check:
Even if a filename passes the regex, the resolved file path is checked to confirm it stays inside the system temp directory. This prevents symlink-based bypass attacks.

Why path traversal is dangerous: If a filename like ../../WEB-INF/web.xml is accepted without validation, the server will read and send the contents of web.xml (which contains DB passwords) to the attacker.

7. Test Area 6 – Hardcoded File Path Removed

File: MoodleController.java – createFile() and createMoodleReportFile()

What was changed:
Before: Files were written to new File("Moodle-Enrolment.csv") (relative path — unpredictable location) and "C:/OnboardingToolExport/" (Windows-only path that may not exist).

After: All file creation uses System.getProperty("java.io.tmpdir") which is the OS-provided temporary directory. On Windows this is C:\Users\<user>\AppData\Local\Temp, on Linux it is /tmp.

Why this matters: The application must run on any server (dev, test, production, cloud). A hardcoded Windows path like C:/OnboardingToolExport/ will crash the application on Linux. A relative path resolves to wherever Tomcat was launched from, which is unpredictable.

8. Test Area 7 – Log File Hardcoded Path Fixed

File: log4j.properties

What was changed:

After: log4j.appender.file.File=${catalina.home}/logs/onboard.log

9. Test Area 8 – Spring Security Schema Version Fixed

File: spring-security.xml

What was changed:
The XML namespace schema declaration was pointing to spring-security-4.0.xsd while the JAR on the classpath was version 4.2.13.RELEASE. Spring Security's namespace handler checks this version and refuses to start if they don't match.

Before: spring-security-4.0.xsd
After: spring-security-4.2.xsd

How to verify: The error BeanDefinitionParsingException: You cannot use a spring-security-4.0.xsd schema with Spring Security 4.2 will no longer appear on startup.

11. Test Area 10 – Spring Framework Version Unified (pom.xml)

File: pom.xml

What was changed:
The project had mixed Spring Framework versions — some dependencies were on 4.2.5.RELEASE and others on 4.3.30.RELEASE. Mixed versions can cause ClassCastException, NoSuchMethodError, or subtle runtime bugs because different modules have different internal APIs.

After: A single property <org.springframework-version>4.3.30.RELEASE</org.springframework-version> is used by all Spring artifacts. 4.3.30 is the final maintenance release of the Spring 4.3.x line.
