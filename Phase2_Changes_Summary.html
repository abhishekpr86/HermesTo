1. Full Migration Roadmap

The migration is split into 7 incremental steps so each can be individually compiled, tested and verified before proceeding.
1
Java 8 → Java 21   DONE
Update java-version property, maven-compiler-plugin to use <release>21</release>, remove obsolete Maven plugins. Fix illegal sun.* imports in source files.
2
Spring 4.3 → 5.3.x + Spring Security 4.2 → 5.8.x   NEXT
Spring 4.x does not support Java 9+ module system officially. Upgrade to Spring 5.3 (last Spring 5 LTS) and Spring Security 5.8 (last pre-Jakarta version). This is the last step before the jakarta namespace change.
3
Log4j 1.x → Logback / SLF4J   PENDING
Log4j 1.2.15 is End-of-Life (2015) and has known CVEs. Replace with Logback (Spring Boot default) and clean up slf4j-log4j12 bridge.
4
Migrate to Spring Boot 3.x Parent POM   PENDING
Add spring-boot-starter-parent 3.x. Brings managed versions for Hibernate 6, HikariCP, Logback, Spring Security 6, etc. Removes need for dozens of individual version declarations.
5
javax.* → jakarta.* Namespace Rename   PENDING
Spring Boot 3 / Jakarta EE 9 renamed all javax.persistence, javax.servlet, javax.mail etc. to jakarta.*. All Java source files and XML configs need updating. Affects every model, controller, and config class.
6
C3P0 → HikariCP Connection Pool   PENDING
C3P0 is EOL and not actively maintained. HikariCP is the Spring Boot 3 default — significantly faster and simpler to configure.
7
XML Configs → Java @Configuration (Optional)   PENDING
Convert servlet-context.xml, spring-security.xml, root-context.xml to Java-based @Configuration classes. IDE-friendly, type-safe, and standard for Spring Boot.

3. pom.xml – Java Version Property

The <java-version> property controls the JDK target for compilation and is also referenced by other plugins.
Reason: The entire Phase 2 goal is to run on Java 21 LTS. Java 21 brings virtual threads, modern GC, pattern matching, and long-term vendor support until 2031. Changing the property alone is not enough — the compiler plugin also needs updating (see next change).
BEFOREAFTER
<java-version>1.8</java-version> <java-version>21</java-version>

4. pom.xml – maven-compiler-plugin

This plugin controls how Maven compiles the Java source files. The old version (2.5.1, released 2012) was incompatible with Java 21.
Reason:
• maven-compiler-plugin 2.5.1 uses com.sun.tools.javac.* internal APIs that are inaccessible on Java 9+.
• The old <source>/<target> pair does not correctly enforce API compatibility — you could target 21 but still accidentally call APIs removed before 21.
• <release>21</release> is the correct modern replacement — it sets source, target, and the bootstrap classpath API check in one flag.
• Version 3.13.0 is the latest stable release fully tested with Java 21.
BEFOREAFTER
<artifactId>maven-compiler-plugin</artifactId> <version>2.5.1</version> <configuration> <source>1.8</source> <target>1.8</target> <compilerArgument>-Xlint:all</compilerArgument> <showWarnings>true</showWarnings> <showDeprecation>true</showDeprecation> </configuration> <artifactId>maven-compiler-plugin</artifactId> <version>3.13.0</version> <configuration> <release>21</release> <compilerArgument>-Xlint:all</compilerArgument> <showWarnings>true</showWarnings> <showDeprecation>true</showDeprecation> </configuration>

5. pom.xml – maven-eclipse-plugin Removed

This plugin was completely removed from the build.
Reason (two problems):

Problem 1 — Causes ExceptionInInitializerError on Java 21:
maven-eclipse-plugin 2.9 (released 2012) uses sun.reflect.* and other private JDK internal APIs. Java 9 introduced the module system which strictly blocks access to these APIs. On Java 21, the plugin's static initializer crashes immediately with:
ExceptionInInitializerError [in thread "Worker-18: Building Project onboarding"]
This makes Eclipse unable to process pom.xml at all, showing the error on line 1.

Problem 2 — Plugin is completely obsolete:
maven-eclipse-plugin was used in Eclipse 2.x/3.x era (pre-2009) to generate .classpath and .project files. Since Eclipse 3.5+, the m2e (Maven to Eclipse) integration handles this automatically. Eclipse (and Spring Tool Suite) no longer needs or uses this plugin. It has not been updated since 2012 and is officially retired.
BEFOREAFTER
<plugin> <artifactId>maven-eclipse-plugin</artifactId> <version>2.9</version> <configuration> <additionalProjectnatures>...</additionalProjectnatures> <additionalBuildcommands>...</additionalBuildcommands> <downloadSources>true</downloadSources> <downloadJavadocs>true</downloadJavadocs> </configuration> </plugin> /* REMOVED ENTIRELY */ Eclipse handles .classpath and .project generation automatically via m2e. No replacement needed.

6. pom.xml – exec-maven-plugin

Used to run the Scheduler main class via Maven. Upgraded to a Java 21 compatible version.
Reason: Version 1.2.1 (2012) uses deprecated internal Maven APIs removed in Maven 3.9+. Version 3.3.0 supports Java 21 and Maven 3.9+.
BEFOREAFTER
<artifactId>exec-maven-plugin</artifactId> <version>1.2.1</version> <artifactId>exec-maven-plugin</artifactId> <version>3.3.0</version>

7. pom.xml – maven-war-plugin

Used to package the application as a WAR file for Tomcat deployment.
Reason: Version 3.0.0 (2016) does not correctly handle the Java 9+ module descriptor (module-info.class) and has a known conflict with Servlet 4.0+ APIs used in Tomcat 9/10. Version 3.4.0 is the latest stable release supporting Java 21 and Tomcat 10. The missing <groupId>org.apache.maven.plugins</groupId> was also added for correctness.
BEFOREAFTER
<!-- groupId missing --> <artifactId>maven-war-plugin</artifactId> <version>3.0.0</version> <groupId>org.apache.maven.plugins</groupId> <artifactId>maven-war-plugin</artifactId> <version>3.4.0</version>

8. Employee.java – Illegal sun.* Import Removed

src/main/java/com/capgemini/onboarding/model/Employee.java — line 36

The error reported was:
The type sun.security.util.Password is not accessible — Employee.java, line 36
Reason:
sun.security.util.* is a private internal JDK API — it was never meant for application code. In Java 8, the JVM silently allowed access to these packages. In Java 9+, the module system (Project Jigsaw) enforces strict encapsulation: any access to sun.* or com.sun.* internal packages that are not explicitly exported causes a compile error.

Investigation showed that Password was never used anywhere in Employee.java — it was a completely dead stale import, likely left over from copy-paste. The fix is simply to remove the import line.

import com.fasterxml.jackson.annotation.JsonIgnoreProperties; import sun.security.util.Password; // ← illegal on Java 9+ @NamedQueries(value = { ... import com.fasterxml.jackson.annotation.JsonIgnoreProperties; // sun.security.util.Password import removed — // never used; illegal under Java 9+ module system @NamedQueries(value = { ...
