2. pom.xml – Dependency Versions

Three separate version issues were fixed in the Maven build descriptor.

Change 1a – Spring Security version corrected

Reason: Version 4.2.22.RELEASE does not exist on Maven Central. Maven resolved it to an incorrect/unavailable artifact, causing startup failures.

BEFOREAFTER
<artifactId>spring-security-core</artifactId> <version>4.2.22.RELEASE</version> <artifactId>spring-security-config</artifactId> <version>4.2.22.RELEASE</version> <artifactId>spring-security-web</artifactId> <version>4.2.22.RELEASE</version> <artifactId>spring-security-ldap</artifactId> <version>4.2.22.RELEASE</version> <artifactId>spring-security-core</artifactId> <version>4.2.13.RELEASE</version> <artifactId>spring-security-config</artifactId> <version>4.2.13.RELEASE</version> <artifactId>spring-security-web</artifactId> <version>4.2.13.RELEASE</version> <artifactId>spring-security-ldap</artifactId> <version>4.2.13.RELEASE</version>

Change 1b – Spring Framework version unified

Reason: The project had a mix of Spring 4.2.5 and 4.3.30 artifacts on the classpath, causing NoSuchMethodError and unpredictable runtime behaviour. Unified to 4.3.30.RELEASE (latest 4.x patch).

BEFOREAFTER
<org.springframework-version>4.2.5.RELEASE</org.springframework-version> (some individual deps pinned to 4.3.30 separately) <org.springframework-version>4.3.30.RELEASE</org.springframework-version> (single property used by all Spring deps)

Change 1c – Compiler deprecation warnings enabled

Reason: Warnings were silently suppressed, hiding use of deprecated APIs. Enabling them assists with future Phase 2 migration.

BEFOREAFTER
<maven-compiler-plugin> config had no showWarnings or showDeprecation flags <compilerArgument>-Xlint:all</compilerArgument> <showWarnings>true</showWarnings> <showDeprecation>true</showDeprecation>

3. spring-security.xml – XML Schema Version

src/main/webapp/WEB-INF/spring/appServlet/spring-security.xml

The XML namespace declaration referenced the wrong schema version. Spring loaded the 4.2 JAR but tried to validate beans against the 4.0 XSD, causing a BeanDefinitionParsingException on every Tomcat startup.

Change 2 – Schema namespace corrected

Reason: Schema version in xsi:schemaLocation must match the Spring Security JAR version on the classpath.

BEFOREAFTER
...spring-security-4.0.xsd ...spring-security-4.2.xsd

4. UserManagementDaoImpl.java – SQL Injection

src/main/java/com/capgemini/onboarding/dao/impl/UserManagementDaoImpl.java

The password change flow used string concatenation to build SQL queries, allowing an attacker to inject SQL via the username or password fields. Replaced with fully parameterised native queries.

Change 3 – SQL injection removed (SELECT + UPDATE)

Reason: OWASP A03 – SQL Injection. String-concatenated SQL is directly exploitable.

BEFOREAFTER
// String concatenation — injectable session.createNativeQuery( "SELECT password FROM users WHERE username = '" + username + "'") .uniqueResult(); session.createNativeQuery( "UPDATE users SET password = '" + hashedNewPassword + "' WHERE username = '" + username + "'") .executeUpdate(); // Parameterised — injection-proof session.createNativeQuery( "SELECT password FROM users WHERE username = :username") .setParameter("username", username) .uniqueResult(); session.createNativeQuery( "UPDATE users SET password = :password, isFirstLogin = '0'" + " WHERE username = :username") .setParameter("password", hashedNewPassword) .setParameter("username", username) .executeUpdate();

5. CommonUtil.java – Password Security

src/main/java/com/capgemini/onboarding/util/CommonUtil.java

Two password-related security issues were fixed in the utility class.

Change 4a – Plaintext password logging removed

Reason: OWASP A02 – Cryptographic Failures. Logging plaintext passwords exposes them in log files, which may be accessible to other users or systems.

BEFOREAFTER
public static String getPasswordBcrypt(String password) { logger.info("getPasswordBcrypt password :" + password); String hashedPassword = passwordEncoder.encode(password); return hashedPassword; } public static String getPasswordBcrypt(String password) { String hashedPassword = passwordEncoder.encode(password); return hashedPassword; }

Change 4b – MD5 hash method removed

Reason: OWASP A02 – Cryptographic Failures. MD5 is cryptographically broken and must not be used for password hashing. BCrypt (already in use) is the correct algorithm. The unused imports BigInteger, MessageDigest, NoSuchAlgorithmException were also removed.

BEFOREAFTER
import java.math.BigInteger; import java.security.MessageDigest; import java.security.NoSuchAlgorithmException; ... public static String getPasswordMD5(String password) throws NoSuchAlgorithmException { MessageDigest md = MessageDigest.getInstance("MD5"); byte[] messageDigest = md.digest(password.getBytes()); BigInteger number = new BigInteger(1, messageDigest); String hashtext = number.toString(16); ... return hashtext; } // Entire getPasswordMD5() method removed. // Unused imports BigInteger, MessageDigest, // NoSuchAlgorithmException also removed.

6. MoodleController.java – Race Condition & Path Traversal

src/main/java/com/capgemini/onboarding/MoodleController.java

This Spring MVC controller is a singleton. Storing request-scoped state in instance fields causes race conditions under concurrent requests. Additionally, file download endpoints accepted user-supplied filenames without validation, enabling path traversal attacks.

Change 5a – Shared instance variables removed (race condition)

Reason: OWASP A01/A04 – Insecure Design. Singleton controllers sharing mutable fields across requests can corrupt data or serve one user's file to another.

BEFOREAFTER
// Singleton-scoped instance fields shared across requests private String absolutePath; private HttpSession session; private String role_id; // Fields removed entirely. // absolutePath is now a local variable in each method. // session obtained from method parameter where needed.

Change 5b – Path traversal protection added to download endpoints

Reason: OWASP A01 – Broken Access Control. Without validation, ../../etc/passwd-style filenames could read arbitrary server files.

BEFOREAFTER
// No validation on user-supplied filename File file = new File("C:/OnboardingToolExport/" + filename); // Whitelist validation – only word chars, hyphens, dots allowed private static final Pattern SAFE_FILENAME = Pattern.compile("[\\w\\-\\.]+"); if (!SAFE_FILENAME.matcher(file).matches()) { response.sendError(SC_BAD_REQUEST, "Invalid filename"); return; } // Canonical path check – belt-and-suspenders File f = new File(System.getProperty("java.io.tmpdir"), filename); if (!f.getCanonicalPath().startsWith( new File(System.getProperty("java.io.tmpdir")) .getCanonicalPath())) { response.sendError(SC_FORBIDDEN); return; }

Change 5c – Hardcoded Windows path removed

Reason: OWASP A05 – Security Misconfiguration. C:/OnboardingToolExport/ fails on non-Windows servers and fails if the directory doesn't exist.

BEFOREAFTER
String excelFilePath = "C:/OnboardingToolExport/"; String excelFilePath = System.getProperty("java.io.tmpdir") + File.separator;

Change 6e – Path traversal protection + temp file cleanup in downloadFile()

Reason: OWASP A01 – Broken Access Control. No validation on the {file} path variable allowed traversal to arbitrary files. Also added automatic deletion of the temp file after download to avoid temp directory accumulation.

BEFOREAFTER
public void downloadFile(@PathVariable("file") String file, ...) { // No validation whatsoever File metricsReportFile = new File("C:/OnboardingToolExport/" + file + ".xlsx"); InputStream is = new FileInputStream(metricsReportFile); FileCopyUtils.copy(is, response.getOutputStream()); // File never deleted — accumulates indefinitely } public void downloadFile(@PathVariable("file") String file, ...) { // Step 1: Reject traversal characters if (file.contains("/") || file.contains("\\") || file.contains("..") || file.contains("\0")) { response.sendError(SC_BAD_REQUEST, "Invalid filename"); return; } File metricsReportFile = new File(System.getProperty("java.io.tmpdir"), file + ".xlsx"); // Step 2: Canonical path must still be inside tmpdir if (!metricsReportFile.getCanonicalPath().startsWith( new File(System.getProperty("java.io.tmpdir")) .getCanonicalPath())) { response.sendError(SC_FORBIDDEN); return; } InputStream is = new FileInputStream(metricsReportFile); FileCopyUtils.copy(is, response.getOutputStream()); response.flushBuffer(); is.close(); // Step 3: Delete temp file after successful download if (metricsReportFile.delete()) { logger.info("After File delete true"); } }

8. log4j.properties – Hardcoded Log Path

src/main/resources/log4j.properties

The log file path was hardcoded to a developer's local machine path, causing silent log loss on any other environment.

Change 7 – Log file path made portable

Reason: OWASP A05 – Security Misconfiguration. A hardcoded path fails silently on other machines; using ${catalina.home} writes logs to the standard Tomcat logs directory on any server.

BEFOREAFTER
log4j.appender.file.File=C:/work/psa/onboarding/logs/onboard.log log4j.appender.file.File=${catalina.home}/logs/onboard.log
