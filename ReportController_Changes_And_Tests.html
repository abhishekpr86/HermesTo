Part 1 – What Was The Problem?

When a user clicked the download button for the Moodle Completion Report, the server threw:

java.io.FileNotFoundException: C:\OnboardingToolExport\MoodleCompletion_14-05-2026 13 42 35.xlsx (The system cannot find the file specified)

Why? The Excel file was being written to the system temp directory (java.io.tmpdir) by MoodleController, but ReportController.downloadFile() was trying to read it from C:\OnboardingToolExport\ — a folder that does not exist on the server. The file was being written to one place and looked up in another.

Additionally, the same C:\OnboardingToolExport\ path was hardcoded in 3 other methods in the same file, meaning ALL report exports would fail on any machine or OS that does not have exactly that folder.

Part 2 – Changes Made (5 Changes)

Change 1 – Removed Shared Instance Variable absolutePath
Location: Class-level field, approximately line 80
Type of issue: Race Condition (concurrent request safety)

Explanation: Spring controllers are singletons — only one instance exists and is shared by ALL users simultaneously. private String absolutePath was a shared field. If User A generated a report and User B simultaneously generated a different report, User B's path would overwrite User A's — and User A's download would try to fetch the wrong file.
BEFORE (vulnerable):
public class ReportController {
    private Logger logger = Logger.getLogger(ReportController.class);
    private String absolutePath;   // ← shared across ALL users — race condition
AFTER (fixed):
public class ReportController {
    private Logger logger = Logger.getLogger(ReportController.class);
    // absolutePath removed — was never needed with tmpdir approach

Change 2 – createResourceReportFile() – Write Path Fixed
Location: Method createResourceReportFile() (mapped to /ResReportExcelExport)
Type of issue: Hardcoded Windows path — would crash on Linux / cloud deployment

Explanation: The Employee List Excel report was written to C:/OnboardingToolExport/. This path only works on a specific Windows machine where the folder was manually created. On any other machine, test server, or Linux host it throws an exception. The OS-provided temp directory (java.io.tmpdir) is always available on every OS.
BEFORE:
String excelFilePath = "C:/OnboardingToolExport/";   // ← hardcoded, Windows-only
AFTER:
String excelFilePath = System.getProperty("java.io.tmpdir") + File.separator;   // ← works on any OS

Change 3 – bisRotationExcelExport() – Write Path Fixed + Relative File Removed
Location: Method bisRotationExcelExport() (mapped to /bisRotationExcelExport)
Type of issue: Hardcoded Windows path + relative file path + race condition via this.absolutePath

Explanation: Two problems existed here:
  • new File("ResourceRotation-Export.xlsx") creates a file relative to wherever Tomcat was launched from — this is unpredictable and differs between machines
  • this.absolutePath = file.getAbsolutePath() wrote to the shared instance field (race condition)
  • "C:/OnboardingToolExport/ResourceRotation-Export.xlsx" is a Windows-only hardcoded path
BEFORE:
File file = new File("ResourceRotation-Export.xlsx");    // ← relative path, unpredictable
System.out.println(file.getAbsolutePath());
this.absolutePath = file.getAbsolutePath();              // ← shared field, race condition
logger.info("absolutePath " + absolutePath);

String excelFilePath = "C:/OnboardingToolExport/ResourceRotation-Export.xlsx";  // ← hardcoded
AFTER:
// All three lines above removed
String excelFilePath = System.getProperty("java.io.tmpdir") + File.separator + "ResourceRotation-Export.xlsx";

Change 4 – downloadFile() – Root Cause Fix + Path Traversal Protection Added
Location: Method downloadFile() (mapped to /downloadReportFile/{file})
Type of issue: FileNotFoundException (direct bug) + Path Traversal attack vulnerability

Explanation of the download bug: The file path variable from the URL was appended directly to C:/OnboardingToolExport/. Since the file was written to tmpdir, it was never at that path — causing the FileNotFoundException reported in the error.

Explanation of path traversal: The file parameter comes from the user's browser URL. Without validation, an attacker could pass ../../WEB-INF/web.xml (which contains database passwords) and the server would send that file to them. Two protection layers were added:
  • Layer 1: Check for /, \, .., and null bytes — return HTTP 400 immediately
  • Layer 2: Canonical path check — even if Layer 1 is somehow bypassed, verify the resolved file path is inside tmpdir
BEFORE (broken + vulnerable):
logger.info("PROD CHECK - Generated excel file by /ResReportExcelExport call.");
logger.info("absolute " + this.absolutePath);
File metricsReportFile = new File("C:/OnboardingToolExport/" + file + ".xlsx");  // ← wrong location, no validation
InputStream is = new FileInputStream(metricsReportFile);   // ← FileNotFoundException thrown here
response.setHeader("Content-Disposition", "attachment; filename=" + metricsReportFile.getName());
FileCopyUtils.copy(is, response.getOutputStream());
response.flushBuffer();
is.close();
AFTER (fixed + secured):
// Layer 1: reject dangerous characters
if (file.contains("/") || file.contains("\\") || file.contains("..") || file.contains("\0")) {
    response.sendError(HttpServletResponse.SC_BAD_REQUEST, "Invalid filename");
    return;
}
// Resolve file inside tmpdir
File metricsReportFile = new File(System.getProperty("java.io.tmpdir"), file + ".xlsx");
// Layer 2: canonical path check — file must stay inside tmpdir
if (!metricsReportFile.getCanonicalPath().startsWith(
        new File(System.getProperty("java.io.tmpdir")).getCanonicalPath())) {
    response.sendError(HttpServletResponse.SC_FORBIDDEN);
    return;
}
logger.info("downloadReportFile: serving " + metricsReportFile.getName());
InputStream is = new FileInputStream(metricsReportFile);
response.setHeader("Content-Disposition", "attachment; filename=" + metricsReportFile.getName());
FileCopyUtils.copy(is, response.getOutputStream());
response.flushBuffer();
is.close();

Change 5 – onbReqReportFile() – Write Path Fixed
